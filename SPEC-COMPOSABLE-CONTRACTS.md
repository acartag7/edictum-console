# Spec: Composable Contracts — Complete Contracts Page Redesign

> **Status:** Draft v2 — 2026-03-03
> **Priority:** Ship-blocker. Lost first customer over this gap.
> **Scope:** Console-only. Zero core library changes.
> **Absorbs:** SPEC-PRODUCT-FEATURES.md §2 (Contract Editor), §3 (Template Library), §4 (AI Chat Wizard).

---

## 1. Problem Statement

The current model treats contract bundles as monolithic YAML files. One bundle per name, deployed to an environment. Every agent with the same `bundle_name` config gets the same contracts.

A team with 50 agents across different departments cannot:
- Define a **baseline** of rules that apply to most agents
- Add **role-specific rules** for subsets (finance, engineering, customer-facing)
- **Reuse** a single contract across multiple bundles
- **Shadow-test** a new rule on one agent group before rolling out to all
- **Change an agent's contracts** from the dashboard without redeploying the agent
- **Author contracts from observed events** — turning a dangerous tool call into a reusable rule
- **Get AI help writing contracts** without memorizing the YAML schema

They are forced to maintain N nearly-identical YAML files with copy-pasted rules. This doesn't scale past 3-5 agents.

## 2. User Stories

### Persona: Platform Engineer (the 50-agent team lead)

> **US-1:** As a platform engineer, I want to author individual contracts and store them in a shared library, so that I don't copy-paste across YAML files.

> **US-2:** As a platform engineer, I want to compose a bundle by selecting contracts from the library and choosing enforce/observe mode per contract, so that each agent role gets exactly the rules it needs.

> **US-3:** As a platform engineer, I want to add a new contract to an existing bundle and have all assigned agents pick it up via hot-reload, without redeploying or restarting any agent.

> **US-4:** As a platform engineer, I want to shadow-test a new contract on one bundle (observe mode) before promoting it to enforce, so that I can see how it would behave without blocking agents.

> **US-5:** As a platform engineer, I want to see which bundles use a given contract, so I understand the blast radius before editing it.

### Persona: CISO / Security Lead

> **US-6:** As a CISO, I want a "baseline security" contract set that is enforced on ALL agents, with no opt-out, so that compliance requirements are met company-wide.

> **US-7:** As a CISO, I want to see a diff of what changed in a bundle before it's deployed, so I can review the impact.

### Persona: AI Engineer (agent developer)

> **US-8:** As an AI engineer, I want my agent to receive its contracts from the server without hardcoding a bundle name, so that the platform team can change my agent's contracts without me changing config.

> **US-9:** As an AI engineer, I want to upload a complete YAML file and have the console extract individual contracts into the library, so I can migrate from local files without rewriting.

> **US-10:** As an AI engineer, I want to create a contract directly from an event I see in the Events Feed (e.g., a dangerous tool call that was allowed), so I can go from "that shouldn't happen" to "it can never happen again" in 30 seconds.

> **US-11:** As an operator who knows what I want to restrict but not the YAML syntax, I want to describe my intent in plain English and get a valid contract generated by AI, so I can create governance rules without memorizing the schema.

## 3. Terminology

| Term | Definition | Analogy |
|------|-----------|---------|
| **Contract** | A single governance rule. Has an `id`, `type`, `when`/`then`. The authoring unit. | A Lego brick |
| **Bundle** | A composed set of contracts assembled for deployment. Versioned, signed. The deployment unit. | A Lego model (assembled from bricks) |
| **Assignment** | A mapping that determines which bundle an agent receives. The routing unit. | Instructions: "this model goes to this shelf" |
| **Contract Library** | The tenant's collection of reusable contracts. | The Lego box |

> **Note:** The core library uses `ContractBundle` as the YAML document kind. That doesn't change. The console adds a layer of individual contract management *on top of* the bundle format.

## 4. Design Principles

1. **Composition is server-side.** Agents receive a single flat bundle YAML. They never know it was composed from pieces. Zero core library changes.

2. **Bottom-up is the primary workflow.** Author contracts individually → compose into bundles → deploy. This is what the customer asked for.

3. **Top-down for migration.** Upload a complete YAML → console extracts individual contracts into the library. Existing users don't start over.

4. **Contracts are immutable versions.** Editing a contract creates a new version. Bundles reference specific contract versions. This gives auditability and safe rollback.

5. **Bundles auto-compose on deploy.** When you deploy a bundle, the server assembles the YAML from the referenced contracts, signs it, and pushes via SSE. No manual YAML construction.

6. **Backward compatible.** Existing `from_server(bundle_name="X")` keeps working. Monolithic uploads keep working (decomposed on import). Phase 2 adds server-side assignment; Phase 1 keeps agent self-selection.

7. **Shared contract updates propagate.** When a contract is updated, all bundles using it can be recomposed and redeployed. The user controls whether this is automatic or manual (per-bundle setting).

8. **AI is optional, never required.** The AI assistant enhances the authoring experience but the console is fully functional without it. No AI provider configured = chat panel hidden, everything else works.

## 5. Data Model

### New Tables

#### `Contract` (the authoring unit)

Individual, reusable governance rules stored in the contract library.

```
contracts
├── id              UUID PK
├── tenant_id       UUID FK → tenants.id
├── contract_id     String          -- e.g., "block-sensitive-reads" (YAML `id` field)
├── version         Integer         -- auto-incremented per (tenant_id, contract_id)
├── type            String          -- "pre" | "post" | "session" | "sandbox"
├── name            String          -- human-readable display name
├── description     Text nullable   -- what this contract does
├── definition      JSON            -- the full contract body (type, when, then, tool, etc.)
├── tags            JSON            -- ["baseline", "security", "pii"] for categorization
├── is_latest       Boolean         -- true for the latest version of this contract_id
├── created_by      String
├── created_at      DateTime(tz)
└── UNIQUE(tenant_id, contract_id, version)
```

**Notes:**
- `contract_id` is the stable identifier (like a package name). `version` increments on each edit.
- `definition` is the JSON representation of the contract body — everything needed to reconstruct the YAML fragment. Includes `type`, `tool`/`tools`, `when`, `then`, `mode`, `enabled`, `limits`, sandbox fields, etc.
- `is_latest` is a denormalized flag for fast queries. Updated atomically when a new version is created. **Use `SELECT ... FOR UPDATE` or a savepoint retry loop** (same pattern as `upload_bundle` in `bundle_service.py`) to prevent concurrent version creates from leaving multiple `is_latest=true` rows.
- `tags` enable filtering/categorization in the library UI ("show me all security contracts", "baseline contracts").

#### `BundleComposition` (the composition recipe)

Defines which contracts make up a bundle, in what order, with what mode overrides.

```
bundle_compositions
├── id              UUID PK
├── tenant_id       UUID FK → tenants.id
├── name            String          -- bundle name (matches Bundle.name)
├── description     Text nullable
├── defaults_mode   String          -- "enforce" | "observe" (bundle-level default)
├── update_strategy  String          -- "manual" | "auto_deploy" | "observe_first" (per-bundle)
├── tools_config    JSON nullable   -- tool side-effect classifications
├── observability   JSON nullable   -- observability config
├── created_by      String
├── updated_at      DateTime(tz)
├── created_at      DateTime(tz)
└── UNIQUE(tenant_id, name)
```

#### `BundleCompositionItem` (contract → bundle membership)

Join table: which contracts are in which bundle, in what order, with what mode.

```
bundle_composition_items
├── id              UUID PK
├── composition_id  UUID FK → bundle_compositions.id
├── contract_id     UUID FK → contracts.id  -- references a specific contract VERSION
├── position        Integer         -- evaluation order within the bundle
├── mode_override   String nullable -- "enforce" | "observe" | null (uses bundle default)
├── enabled         Boolean         -- can disable a contract within a bundle without removing it
├── created_at      DateTime(tz)
└── UNIQUE(composition_id, contract_id)
```

**Notes:**
- `contract_id` FK points to a specific `contracts.id` (a specific version). When a contract is updated, the bundle can either auto-update to the latest version or stay pinned.
- `position` determines evaluation order. Lower = evaluated first. Gaps allowed (10, 20, 30) for easy insertion.
- `mode_override` lets you enforce a contract in one bundle and observe it in another (the shadow-test workflow).
- **CRITICAL: Tenant isolation on FK.** The `contract_id` FK is a UUID pointing to `contracts.id`, which is not tenant-scoped at the DB constraint level. The **service layer MUST verify** that every referenced contract belongs to the same `tenant_id` as the composition. This check runs in the assembly algorithm (§6 step 3a) and in the composition create/update endpoints (§7). Adversarial tests for C2 (Appendix C) verify this boundary.

#### `TenantAiConfig` (AI assistant configuration)

Per-tenant AI provider settings. Stored in DB (not env vars) for UI-managed configuration.

```
tenant_ai_configs
├── id                  UUID PK
├── tenant_id           UUID FK → tenants.id (UNIQUE)
├── provider            String          -- "anthropic" | "openai" | "openrouter" | "ollama" | "none"
├── api_key_encrypted   Bytes nullable  -- encrypted with EDICTUM_SECRET_KEY via NaCl SecretBox
├── model               String nullable -- null = provider default
├── base_url            String nullable -- null = provider default (OpenRouter users set this)
├── updated_at          DateTime(tz)
├── updated_by          String
└── UNIQUE(tenant_id)
```

**Notes:**
- API key encrypted at rest using the same NaCl SecretBox pattern as signing keys (`EDICTUM_SECRET_KEY`).
- API key **never returned in API responses** — GET returns masked value (`sk-...••••`).
- Env vars (`EDICTUM_AI_PROVIDER`, `EDICTUM_AI_API_KEY`, `EDICTUM_AI_MODEL`, `EDICTUM_AI_BASE_URL`) serve as **fallback only** for Docker pre-configuration. DB config takes priority. If neither exists, AI features are disabled.
- **Tenant isolation:** Each tenant has their own config. Tenant A's API key is never visible to or usable by Tenant B. The `assist` endpoint reads the calling tenant's config only.

#### `AgentRegistration` (Phase 2 — persistent agent identity)

> Phase 2. Not in MVP. Included here for data model completeness.

```
agent_registrations
├── id              UUID PK
├── tenant_id       UUID FK → tenants.id
├── agent_id        String          -- matches X-Edictum-Agent-Id header
├── display_name    String nullable -- human-friendly name
├── tags            JSON            -- {"role": "finance", "team": "payments"}
├── bundle_name     String nullable -- explicit assignment (overrides pattern matching)
├── last_seen_at    DateTime(tz) nullable
├── created_at      DateTime(tz)
└── UNIQUE(tenant_id, agent_id)
```

#### `AssignmentRule` (Phase 2 — pattern-based routing)

> Phase 2. Not in MVP.

```
assignment_rules
├── id              UUID PK
├── tenant_id       UUID FK → tenants.id
├── priority        Integer         -- lower = evaluated first
├── pattern         String          -- agent_id glob pattern (e.g., "finance-*")
├── tag_match       JSON nullable   -- {"role": "finance"} — all tags must match
├── bundle_name     String          -- which bundle to assign
├── env             String          -- which environment this rule applies to
├── created_at      DateTime(tz)
└── UNIQUE(tenant_id, priority)
```

### Changes to Existing Tables

#### `Bundle` table — add columns (requires Alembic migration)

The existing `Bundle` table continues to store the **assembled** YAML (the deployment artifact). New nullable columns:

```
bundles (existing table, add columns)
├── composition_id  UUID FK nullable → bundle_compositions.id
├── composition_snapshot JSON nullable -- snapshot of contract versions used (for audit)
```

- `composition_id`: links to the composition recipe that produced this bundle version. Null for monolithic uploads (backward compat). **ON DELETE SET NULL** — deleting a composition does not affect deployed bundles.
- `composition_snapshot`: JSON array of `[{contract_id, version, mode}]` — the exact contracts that were assembled into this bundle. Immutable snapshot for auditability.

**Alembic migration required.** New tables are additive, but the two new columns on `bundles` require a schema migration. No data migration needed — existing rows get `null` for both columns.

#### `Deployment` table — no changes

Deployments continue to reference `(bundle_name, bundle_version)`. The composition is resolved at bundle creation time, not at deployment time.

## 6. Composition Algorithm

### Assembly (Server-Side)

When a bundle is composed from a `BundleComposition`:

```
1. Load BundleComposition (name, defaults_mode, tools_config, observability)
2. Load BundleCompositionItems ordered by position ASC
3. For each item where enabled=True:
   a. Load the referenced Contract version
   b. VERIFY contract.tenant_id == composition.tenant_id (CRITICAL: tenant isolation)
   c. Apply mode_override if set, else use contract's own mode, else bundle defaults_mode
   d. Add to contracts list
4. Construct a valid ContractBundle YAML:
   apiVersion: edictum/v1
   kind: ContractBundle
   metadata:
     name: {composition.name}
     description: {composition.description}
   defaults:
     mode: {composition.defaults_mode}
   contracts: [{assembled contracts in position order}]
   tools: {composition.tools_config or omit}
   observability: {composition.observability or omit}
5. Validate the assembled YAML against the core library's schema (import from core)
6. Compute revision_hash = SHA-256(yaml_bytes)
7. Create a new Bundle row with the assembled YAML + composition_snapshot
8. Sign if deploying
```

### Contract ID Uniqueness

Within a composed bundle, contract `id` values must be unique. The composition UI enforces this — you cannot add two contracts with the same `contract_id` to the same bundle. If the same rule needs different configs for different contexts, create a variant with a different `id`.

**Exception: `observe_first` shadow contracts** — see §6.4 below.

### Mode Resolution

Priority order (highest to lowest):
1. `BundleCompositionItem.mode_override` — explicit per-contract-per-bundle override
2. `Contract.definition.mode` — the contract's own mode
3. `BundleComposition.defaults_mode` — the bundle-level default

### Shared Contract Updates — Update Strategy

When a contract gets a new version, the behavior depends on the bundle's `update_strategy`. Teams choose the strategy that fits their risk tolerance. This is configurable per-bundle.

**`manual`** (safest — default):
1. Dashboard shows an "updates available" indicator on the bundle
2. User reviews the diff (old contract version vs new)
3. User chooses one of:
   - **Deploy now** — update contract version, re-assemble, deploy immediately
   - **Deploy as observe** — add the new version in observe mode alongside current. Shadow-test before promoting.
   - **Dismiss** — skip this update, stay on pinned version

**`observe_first`** (recommended for teams iterating fast):
1. New contract version is automatically added to the bundle as a shadow contract (see §6.4)
2. Dashboard shows the shadow contract's observe results (would-deny events, etc.)
3. User reviews shadow data and either:
   - **Promote** — replace the old version with the new one in enforce mode, remove shadow
   - **Reject** — remove the shadow version, keep the old enforced version
4. This uses the `.obs` suffix convention described in §6.4

**`auto_deploy`** (fastest, for teams with strong CI/testing):
1. New contract version automatically replaces the old one
2. Bundle is re-assembled and deployed immediately
3. SSE pushes `contract_update` to all connected agents
4. Useful for observe-only bundles or teams with rollback confidence

### 6.4 Shadow Contracts — `observe_first` and `.obs` Suffix

When `observe_first` triggers, the assembly algorithm adds the new contract version alongside the old one using a `.obs` suffix to avoid ID collision:

**In the assembled YAML:**
```yaml
contracts:
  - id: block-sensitive-reads          # v1, enforce (current pinned version)
    type: pre
    mode: enforce
    ...
  - id: block-sensitive-reads.obs      # v2, observe (shadow-testing)
    type: pre
    mode: observe
    ...
```

**Rules:**
- The `.obs` suffix is added by the assembly algorithm, not stored in the DB.
- The shadow item is a second `BundleCompositionItem` pointing to the new contract version, with `mode_override=observe` and `enabled=true`.
- Events from shadow contracts show `decision_name: "block-sensitive-reads.obs"` in the Events Feed — useful for identifying shadow evaluations.
- **Promote:** Remove the old composition item, update the new item's `mode_override` to the desired mode, re-assemble (the `.obs` suffix is dropped because there's no longer a conflict).
- **Reject:** Remove the shadow composition item, re-assemble.
- The `.obs` suffix is deterministic and stripped during promote — no user intervention on naming.

## 7. API Design

### New Endpoints: Contract Library

```
# List contracts in the library
GET /api/v1/contracts
  ?type=pre|post|session|sandbox  (optional filter)
  ?tag=security                   (optional filter)
  ?search=pii                     (optional full-text search on name/description)
Response: ContractSummary[]

# Get contract details (latest version)
GET /api/v1/contracts/{contract_id}
Response: ContractDetail (includes version history)

# Get specific contract version
GET /api/v1/contracts/{contract_id}/versions/{version}
Response: ContractDetail

# Create a new contract
POST /api/v1/contracts
Body: {
  "contract_id": "block-sensitive-reads",
  "name": "Block Sensitive Data Reads",
  "description": "Prevents agents from reading PII-tagged columns",
  "type": "pre",
  "definition": { ... },  // the contract body
  "tags": ["security", "pii", "baseline"]
}
Response: ContractDetail

# Update a contract (creates new version)
PUT /api/v1/contracts/{contract_id}
Body: { "definition": { ... }, "description": "..." }
Response: ContractDetail (new version)

# Delete a contract (only if unused by any BundleCompositionItem)
DELETE /api/v1/contracts/{contract_id}
Response: 204 | 409 (conflict if referenced by any composition)

# Import contracts from a YAML bundle (top-down decomposition)
POST /api/v1/contracts/import
Body: { "yaml_content": "..." }
Response: {
  "contracts_created": [...],
  "contracts_updated": [...],   // existing contract_ids → new version created
  "bundle_composition_created": "bundle-name"  // auto-creates composition
}

# List bundles that use this contract
GET /api/v1/contracts/{contract_id}/usage
Response: BundleCompositionSummary[]
```

**Import contract_id derivation:** Uses the YAML `id` field directly. The core library requires `id` on every contract, so valid YAML always has it. If a `contract_id` already exists in the library (same tenant), the import creates a **new version** (not an error). The response distinguishes `contracts_created` vs `contracts_updated` so the user knows what happened.

### New Endpoints: Bundle Composition

```
# List bundle compositions
```

```python
class AIProvider(ABC):

    async def stream_response(
        self,
        messages: list[dict],

```

**System prompt** embeds the full edictum contract schema: all 4 contract types (pre, post, session, sandbox), 13 selectors, 15 operators, constraints, and a complete working example. ~4K tokens of reference material from the core library's `yaml-reference.md`.


### Tenant Isolation

All new endpoints filter by `tenant_id`. The `contracts`, `bundle_compositions`, and `tenant_ai_configs` tables have `tenant_id` columns with FK constraints. Same security model as all other tables.

**Cross-FK tenant check:** When creating/updating a `BundleCompositionItem`, the service layer verifies every referenced `contract_id` belongs to the same tenant as the composition. This is not enforceable at the DB FK level (the FK is on `contracts.id`, a UUID PK). The service layer is the enforcement point. Return 404 (not 403) for cross-tenant references.

## 9. UI Design — Contracts Page (4 Tabs)

The current Contracts page (tabs: Contracts, Versions, Diff, Evaluate) is replaced with a new 4-tab structure that incorporates composable contracts, the contract editor, templates, and AI assistance.

```
[ Library ]  [ Bundles ]  [ Deployments ]  [ Evaluate ]
```

### Tab 1: Library — "What contracts exist?"

**Layout:** Card grid or table with search/filter bar. Templates section at top for new tenants.

**Templates section** (collapsible, shown when tenant has < 3 contracts):
- "Starter Packs" row with 3-4 pre-built bundles (Research Agent, DevOps Agent, Production Governance)
- Each starter pack shows: name, description, contract count, type badges
- "Preview" opens a dialog showing the YAML + contract list
- "Use" imports all contracts into the library + creates a composition. The template YAML is static JSON shipped with the frontend (no backend needed for templates).
- Section collapses/hides once the tenant has their own contracts.

## 10. Migration Path

### Existing Monolithic Bundles → Composed Model

Users with existing bundles have two paths:

**Path A: Import & Decompose (recommended)**
1. User clicks "Import to Library" on an existing bundle
2. Console extracts each contract from the YAML into individual `Contract` rows
3. Uses the YAML `id` field as `contract_id`. If a `contract_id` already exists, creates a new version.
4. Console creates a `BundleComposition` with the same name, referencing all extracted contracts
5. The existing deployed bundle continues to work (it's the same YAML)
6. Now the user can edit individual contracts and compose new bundles

**Path B: Keep Monolithic (backward compat)**
1. Users continue uploading complete YAML files via `POST /api/v1/bundles`
2. These create `Bundle` rows without a `composition_id` (null)
3. Everything works exactly as before
4. User can migrate to composed model later

### Schema Migration

**Alembic migration required.** Changes:
1. New tables: `contracts`, `bundle_compositions`, `bundle_composition_items`, `tenant_ai_configs` (purely additive)
2. New nullable columns on existing `bundles` table: `composition_id`, `composition_snapshot`

No data migration needed — existing rows get `null` for the new columns.

### API Backward Compatibility

- `POST /api/v1/bundles` — unchanged. Monolithic upload still works.
- `GET /api/v1/bundles/{name}/current` — unchanged. Returns assembled YAML regardless of how it was created.
- `GET /api/v1/stream` — unchanged. `contract_update` events include the same fields.
- All existing SDK versions continue to work without any changes.

## Appendix B: Composition Assembly — Pseudocode

```python
async def assemble_bundle(
    db: AsyncSession,
    composition: BundleComposition,
) -> tuple[bytes, list[dict]]:
    """Assemble a BundleComposition into valid ContractBundle YAML.

    Returns (yaml_bytes, composition_snapshot).
    """
    items = await db.execute(
        select(BundleCompositionItem, Contract)
        .join(Contract, BundleCompositionItem.contract_id == Contract.id)
        .where(BundleCompositionItem.composition_id == composition.id)
        .where(BundleCompositionItem.enabled == True)
        .order_by(BundleCompositionItem.position)
    )

    contracts_yaml = []
    snapshot = []
    seen_ids = set()

    for item, contract in items:
        # CRITICAL: tenant isolation — verify every contract belongs to this tenant
        if contract.tenant_id != composition.tenant_id:
            raise ValueError(
                f"Contract {contract.contract_id} belongs to a different tenant"
            )

        contract_id_in_yaml = contract.contract_id
        # Shadow contracts for observe_first get .obs suffix
        if contract.contract_id in seen_ids:
            contract_id_in_yaml = f"{contract.contract_id}.obs"
        seen_ids.add(contract.contract_id)

        # Build the contract YAML dict
        contract_def = dict(contract.definition)
        contract_def["id"] = contract_id_in_yaml

        # Apply mode override
        if item.mode_override:
            contract_def["mode"] = item.mode_override

        contracts_yaml.append(contract_def)
        snapshot.append({
            "contract_id": contract.contract_id,
            "version": contract.version,
            "mode": item.mode_override or contract_def.get("mode"),
            "shadow": contract_id_in_yaml != contract.contract_id,
        })

    # Construct the full bundle
    bundle = {
        "apiVersion": "edictum/v1",
        "kind": "ContractBundle",
        "metadata": {
            "name": composition.name,
        },
        "defaults": {
            "mode": composition.defaults_mode,
        },
        "contracts": contracts_yaml,
    }

    if composition.description:
        bundle["metadata"]["description"] = composition.description
    if composition.tools_config:
        bundle["tools"] = composition.tools_config
    if composition.observability:
        bundle["observability"] = composition.observability

    # Validate against schema (fail-fast before signing/deploying)
    yaml_bytes = yaml.safe_dump(bundle, sort_keys=False).encode()
    load_bundle_string(yaml_bytes.decode())  # raises on invalid

    return yaml_bytes, snapshot
```

## Appendix C: Security Considerations

### New Adversarial Test Categories

| # | Boundary | Risk |
|---|----------|------|
| C1 | Contract tenant isolation | Cross-tenant contract read/write |
| C2 | Composition tenant isolation | Referencing another tenant's contract in a composition |
| C3 | Import decomposition injection | Malicious YAML that creates unexpected contracts |
| C4 | Contract version tampering | Modifying a contract version after bundle deployment |
| C5 | Auto-redeploy safety | Unintended enforce-mode changes via auto-update |
| C6 | AI config tenant isolation | Reading/using another tenant's AI API key |
| C7 | AI assist prompt injection | Malicious messages attempting to leak system prompt or tenant data |

### Tenant Isolation on New Endpoints

Every new endpoint must:
1. Filter by `tenant_id` in all queries
2. Verify FK references are within the same tenant (composition can't reference another tenant's contract — enforced in service layer, not just DB)
3. Return 404 (not 403) for cross-tenant resources (prevent enumeration)
4. AI config: never return raw API keys, never allow tenant A to use tenant B's provider config

### Composition Validation

Before deploying a composed bundle:
1. Validate assembled YAML against the core library's schema
2. Check all contract `id` values are unique within the bundle (`.obs` suffix handles shadow contracts)
3. Check all referenced contracts belong to the same tenant (service layer enforcement)
4. Check no circular references (not applicable in current design, but guard against future changes)

### AI Assistant Security

1. **System prompt is not tenant data** — it contains the contract schema, not tenant-specific info. Safe to share across tenants.
2. **User messages are tenant-scoped** — the `assist` endpoint reads the calling tenant's AI config. Messages are not persisted (stateless, multi-turn via client-side message history).
3. **Prompt injection** — AI responses may contain YAML that looks valid but has unexpected behavior. Client-side validation before "Apply" catches schema violations. Server-side validation at deploy time catches anything the client misses. The AI never auto-deploys — user always reviews before deploy.
4. **API key security** — encrypted at rest, never logged, never in API responses, masked in GET.
